================================================================================
  *** GOLDEN RULE: ALWAYS VERIFY YOUR DATA SOURCE ***
================================================================================

  In timing analysis, knowing WHERE your data comes from is CRITICAL.
  A slack number is meaningless if you don't know which model produced it.

  BEFORE running ANY timing query, ALWAYS run with -debug first:

    pt_client.pl -m latest_gfcn2clienta0_bu_prp_func.max_high.T_85.typical \
      -c 'get_designs' -debug

  CHECK these lines in the output:
    -I- MODEL: .../GFC_CLIENT_26ww08b_...-FCT26WW11B_dcm_daily-...
                                          ^^^^^^^^^^
                                          THIS is your data source tag

    -I- Date modela : '1772976588' at @ Sun Mar  8 06:29:48 2026
    -I- Date modelb : '1773068403' at @ Tue Mar 10 05:21:01 2026
        → Server auto-picks the NEWER model (modelb in this case)

  WHY THIS MATTERS:
    • A fix that looks good on WW10 data might already be solved in WW11
    • A path that was clean last week might be failing this week
    • RCOs, TIPs, and clock changes between builds can shift slack ±50ps
    • Comparing results across different models gives WRONG conclusions

  ALWAYS DOCUMENT:
    When reporting timing results, always note the model tag:
      "RSMOClearVM803H slack = -71.6ps (FCT26WW11B, func.max_high.T_85.typical)"

================================================================================
  1. WHAT IS pt_client.pl?
================================================================================

  pt_client.pl is a Perl script that sends PT commands to a LIVE PrimeTime
  server running 24/7 with pre-loaded design databases (DBS). Instead of:
    1. Opening a PT session (5-15 min startup)
    2. Loading a design (10-30 min for core_client)
    3. Running your command
    4. Closing the session

  You just run ONE command and get results in seconds:
    pt_client.pl -m <corner> -c '<PT command>'

  Author: Ooi, Thean Wui (2016), maintained by FCT team (yifrach, ogivol)

================================================================================
  3. BASIC USAGE
================================================================================

  SYNTAX:
    pt_client.pl -m <mode> -c '<PrimeTime command>'

  OPTIONS:
    -m <mode>    Corner/scenario to query (default: func_max)
    -c '<cmd>'   PrimeTime TCL command to execute (MUST be in quotes)
    -o <file>    Save output to file instead of stdout (optional)
    -debug       Show server mapping debug info
    -help/-h     Show usage

  EXAMPLES:

  # Basic timing query
  pt_client.pl -m latest_gfcn2clienta0_bu_prp_func.max_high.T_85.typical \
    -c 'report_timing -through icore0/par_ooo_int/rsmoclearvm803h -max_paths 1 -nosplit'

  # Save to file
  pt_client.pl -m latest_gfcn2clienta0_bu_prp_func.max_high.T_85.typical \
    -c 'report_timing -through icore0/par_ooo_int/rsmoclearvm803h -max_paths 6' \
    -o rsmoclearvm803h_timing.rpt

  # Multiple corners (comma-separated)
  pt_client.pl -m latest_gfcn2clienta0_bu_prp_func.max_high.T_85.typical,latest_gfcn2clienta0_bu_prp_func.max_nom.T_85.typical \
    -c 'report_timing -through icore0/par_ooo_int/rsmoclearvm803h -max_paths 1'

================================================================================
  4. MODE/CORNER NAMING CONVENTION
================================================================================

  Mode format: <version>_<product>_<flow>_<scenario>

  VERSION PREFIX:
    latest_  → most recent model (auto-selects newer of modela/modelb)
    prev_    → previous model
    modela_  → explicitly model A
    modelb_  → explicitly model B

  PRODUCT:
    gfcn2clienta0  → GFC N2 Client A0

  FLOW:
    bu_prp  → BU postCTS (PrimeTime with real parasitics)
    fcl     → FCL (flat chip level, less common)

  SCENARIO (same as STA corners):
    func.max_high.T_85.typical       → Setup, slow process, 85°C (PRIMARY)
    func.max_nom.T_85.typical        → Setup, nominal
    func.max_low.T_85.typical        → Setup, low voltage
    func.max_med.T_85.typical        → Setup, medium
    func.min_low.T_85.typical        → Hold, fast process
    func.min_nom.T_85.typical        → Hold, nominal
    func.max_fast.F_125.rcworst_CCworst_T → Fast corner with RC worst
    fresh.min_fast.F_125.rcworst_CCworst  → Fresh/burn-in min
    fresh.min_hvqk.F_125.rcworst_CCworst  → HVQK qualification
    fresh.min_fast_cold.F_M40.rcworst_CCworst → Cold temperature
    fresh.min_lo_hi_hi.T_85.typical  → Lo-hi-hi corner

  CCworst SUFFIX:
    _CCworst  → With crosstalk (SI) analysis
    (no suffix) → Without crosstalk

  MOST COMMON MODES FOR OUR WORK:
    latest_gfcn2clienta0_bu_prp_func.max_high.T_85.typical   ← SETUP (primary)
    latest_gfcn2clienta0_bu_prp_func.min_low.T_85.typical    ← HOLD
    latest_gfcn2clienta0_bu_prp_func.max_nom.T_85.typical    ← SETUP nominal

================================================================================
  8. FORBIDDEN COMMANDS (read-only server)
================================================================================

  The PT server is READ-ONLY. These commands are BLOCKED:
    set_           (set_attribute, set_load, etc.)
    create_        (create_clock, create_bound, etc.)
    source         (cannot source TCL scripts)
    save           (cannot save session)
    exec           (cannot run shell commands)
    exit/quit      (cannot close server)
    size_cell      (cannot resize cells)
    insert_buffer  (cannot insert buffers)
    fix_           (no ECO commands)
    update_        (no update commands)

  YOU CAN ONLY: report_timing, get_*, sizeof_collection, report_cell,
                report_clock_timing, report_constraint, etc.

================================================================================
  8b. ILLEGAL CELLS CHECK
================================================================================

  ⚠️ When analyzing path reports, always cross-check cell ref_names against
  the illegal cells list. PT may use banned cells for optimization — the
  timing report looks good but the cell will be rejected in physical design.

  Illegal list: $wa/project/GFCN2CLIENTA0/illegal.txt  (~15,746 cells)

  Banned suffixes to watch for:
    CPDELVT  → BANNED (eLVT)
    CPDLVTLL → BANNED (LVT low-leakage)

  Quick check after getting a path report:
    # Save report, then grep for banned cells
    grep -oP '\w+BWP156H\w+' /tmp/report.txt | sort -u | while read c; do
      grep -q "$c" $wa/project/GFCN2CLIENTA0/illegal.txt && echo "ILLEGAL: $c"
    done

  If found: the real slack is WORSE than reported (typically 5-10ps per
  illegal cell). Account for this when estimating fix potential.

  10. estimate_eco — CHECK IF DATA-PATH ECO CAN HELP (PBA mode)
================================================================================

  estimate_eco tells you: can we improve slack by sizing/buffering cells?
  USE THIS BEFORE assuming ECO gains — it prevents wrong estimates.

  IMPORTANT: Our bus analysis uses PBA (path-based analysis).
  estimate_eco reports GBA slack internally, but the DELTA applies to PBA too.
  NEVER use GBA slack values directly — only use the improvement delta.

  10.1 SYNTAX
  -----------

  SIZE_CELL — can we upsize or Vt-swap a cell?

    pt_client.pl -m <corner> -c 'estimate_eco -type size_cell \
      -sort_by slack -nosplit [get_cells <cell_instance>]'

  INSERT_BUFFER — would adding a buffer on a net help?

    pt_client.pl -m <corner> -c 'estimate_eco -type insert_buffer \
      -sort_by slack -nosplit \
      -lib_cells {BUFFD4BWP156HNPPN3P48CPDULVT BUFFD8BWP156HNPPN3P48CPDULVT} \
      [get_pins <pin>]'

  10.2 READING THE OUTPUT
  -----------------------

  The output is sorted by slack (best first):

    lib cell                            area stage_delay  arrival    slack
    --------------------------------------------------------------------------
   *current_cell/XYZD2BWP...CPDULVT    0.097    8.387    927.313   -54.542
    alternative/XYZD4BWP...CPDULVT      0.150    7.200    928.500   -55.800

  * = current cell (already placed)
  If * is first = current cell is ALREADY BEST — no ECO helps
  If alternative is first = that cell would improve slack

  DELTA = alternative_slack - current_slack
    Positive delta = improvement
    Negative delta = makes it worse

  For insert_buffer:
    *No_buffer = current state (no buffer)
    If No_buffer is first = buffer insertion makes it WORSE

  10.3 COMMON FINDINGS
  --------------------

  When ALL cells show * as best:
    -> Path is fully optimized by P&R, no data-path ECO possible
    -> Reason: larger cells have higher input cap, which slows the driver
       more than the cell speeds up (diminishing returns on timing-critical paths)

  When DULVTLL or LVT cells appear on path:
    -> Vt swap to DULVT may help (check estimate_eco to verify)
    -> But on GFC Client A0, most critical paths are already DULVT

  10.4 WORKFLOW: VERIFY ECO GAINS BEFORE CLAIMING THEM
  ----------------------------------------------------

  WRONG approach:
    1. See D1/D2 cells on path
    2. Assume "upsize D1->D4 = ~5ps gain"    <- WRONG! Often WORSE
    3. Claim ECO fixes N paths                <- OVERESTIMATE

  RIGHT approach:
    1. Get worst path: report_timing -through ... -max_paths 1
    2. For each small cell: estimate_eco -type size_cell [get_cells ...]
    3. If * is best: NO ECO helps, move on
    4. If alternative is better: record actual delta
    5. Also try: estimate_eco -type insert_buffer on long nets
    6. Sum verified deltas = real ECO potential

  10.5 EXAMPLE: dcl1rddatam408h BUS (WW15 finding)
  -------------------------------------------------

  DISCOVERY: All 18 top SFFs (810/1024 paths) are fully optimized.
  estimate_eco -type size_cell shows *current = BEST for every cell.
  Only insert_buffer on 1 long net gave +3.9ps (marginal).

  Conclusion: data-path ECO = ~0ps gain. Useful skew push REQUIRED.

  This is EXPECTED for paths through ECC hmatrix + correction logic:
    - Deep XOR/XNR chains (6-8 stages)
    - All cells already DULVT (fastest Vt)
    - Cells already optimally sized by Innovus
    - Net delays dominated by wire length, not cell speed

  10.6 TIPS
  ---------

  - Always run with -sort_by slack — shows best option first
  - Always use -nosplit for cleaner output through pt_client
  - Check BOTH max_rise and max_fall — worst may differ
  - For cross-partition paths: only ECO cells in YOUR partition
  - estimate_eco is READ-ONLY — safe to run on pt_client server
  - Run on the SAME corner as your bus report (func.max_high for setup)
  - For hold paths: use -min flag with func.min_low corner
