=== WHAT IS FCN? ===

  FCN = Front-end Collateral Netlist release.
  It takes RTL -> synthesizes netlists for all 12 partitions -> compares
  against a REF (reference) run -> generates quality indicators -> promotes
  collaterals (Verilog, UPF, FDR, HIPs) for downstream BE (backend) use.

  This is UPSTREAM of what you do as SIO. The FCN release produces the
  netlists that Gil's daily builds (FCT) then use for timing closure.

  Flow: RTL Model -> FCN (netlist) -> FCT (timing) -> SIO (closure)

=== FCN vs FCT ===

  FCN (Amir's flow)              FCT (Gil's flow)
  -------------------------      -------------------------
  RTL -> Netlist synthesis       Netlist -> Place & Route -> STA
  Cadence Genus (partitions)     Synopsys FC (top) + PT (timing)
  Synopsys FC (top/subfc)        PrimeTime for all corners
  Produces .v, .upf, FDR, HIPs  Produces timing XMLs, IFC reports
  ~20 min per partition          ~hours per corner
  Owner: Amir (amenashe)         Owner: Gil

================================================================================
  FULL FCN FLOW — STEP BY STEP
================================================================================

  --- STEP 0: SETUP ---

  Setup command:
    /nfs/site/proj/hdk/pu_tu/prd/liteinfra/1.19.p1/commonFlow/bin/cth_psetup \
      -proj gfc_n2_client/GFC_TS2025.17 -nowash -cfg gfcn2clienta0.cth \
      -ward <your_WA> -x '$SETUP_R2G'

  To see setup command from existing WA:
    cat $WA/sd.reopen

  --- STEP 1: PREPARE NEW RUN ---

  1a. Copy scripts from REF WA:
      cp -rf $REF_WA/fcn_scripts/ .

  1b. Edit fcn_scripts/fcn_user_vars.tcl — update these key ivars:
      - RTL model       -> from Basel's mail
      - HIP tags        -> from Maxim's mail
      - Clock tag       -> from GFC TD daily mail (Haya releases)
                           *** Take latest BU_FCN clock tag (NOT prop) ***
      - Top IO tag      -> from timing collateral, take latest top IO

  1c. Generate Black Box list:
      fcn_xml_hips_vs_rtl_diff <HIPs_TAG> <RTL_MODEL>

      Example:
        fcn_xml_hips_vs_rtl_diff GFCN2SERVERA0V20_25ww24_Wed \
          /nfs/site/proj/gfc/gfc.models.41/core/core-gfc-a0-master-25ww26c \
          | l > fcn_xml_hips_vs_rtl_diff_output.rpt

      Extract BB list:
        l fcn_xml_hips_vs_rtl_diff_output.rpt | g "EBB not in XML" \
          | sed 's/EBB not in XML://' | sort | sed 's/$/\\/' | l

  1d. Change REF symlink if needed:
      rm ref
      ln -s <ref_wa> ref

  --- STEP 2: fcn_prepare -run ---

  What it does:
    - Runs prepare_hip at core level (generates hip_ivar.tcl)
    - Brings all data required for synthesis (under release/latest/)
    - Submits farm jobs for all 12 partitions (Cadence Genus)
    - Also submits Synopsys FC jobs for top-level (icore, core_client)

  Logs: fcn_logs/prepare.log*
  Check: Search for "does not exist" in prepare.log

  Then run: $ward/run.all.csh (or use -run flag)
  Result: Netlists WITHOUT FDRs

  --- STEP 3: MONITOR ---

  Monitor status:
    fcn_status -wait

  Mid-run checkers:
    CDNS: fcn_cdns_check_genus_map
          (checks all partitions finished genus_map)
    SNPS: fcn_snps_check_init_floorplan
          (checks all partitions finished init_floorplan)

  All logs under: fcn_logs/job.*.<cdns/snps>.log

  --- STEP 4: fcn_prepare -run_fdr ---

  What it does:
    - Runs core_client job: builds top flat timing model,
      generates IO constraints for partitions
    - IO constraints file: release/latest/sio_timing_collateral/
    - Then runs 12 jobs (one per partition) of setup_timing (apr_fc)
      consuming the IO constraints
    - FDRs are ready after this step

  FDR reports:
    runs/<PAR>/n2p_htall_conf4/apr_fc/reports/setup_timing2/<PAR>.fdr.rpt

  --- STEP 5: fcn_promote -no_check -tag <TAG> ---

  What it does:
    - Copies synthesized netlists to release/latest/fe_collateral/
    - Creates $ward/fcn_commit.csh for final commit
    - Generates comparison reports (TST vs REF)

  Logs: fcn_logs/promote.log*

  --- STEP 6: REVIEW ---

  6a. Send summary email:
      fcn_mail_summary (alias)
      -> Sends FCN summary to your mail — review before proceeding

  6b. Generate ports diff report:
      fcn_model_comp_tst_vs_ref_client
      -> Results under $WA/core...
      -> Add report + table to the release email

  6c. Run FEV check:
      fcn_promote -run_fev
      Log: fcn_logs/job.core_client.fev.log
      Debug: debug_fev $ward/runs/core_client/1278.6/fev_conformal/fev_fcl/checkpt_flat.lec

  --- STEP 7: PRE-COMMIT CHECKS ---

  Must be clean before committing:
    g -- -E- fcn_promote.log
    g -- -W- fcn_promote.log
    g -- -W- fcn_reports/run_check_latest/*
    g -- -E- fcn_reports/run_check_latest/*
    g -- -E- fcn_reports/csv_latest/*

  -E- (errors) in fcn_promote.log = FATAL, must fix
  -W- (warnings) = not fatal but should be clean by end of project

  --- STEP 8: COMMIT ---

  $ward/fcn_commit.csh
    -> Commits fe_collateral/ and fdr_collateral/ to $PROJ_ARCHIVE

  --- FULL RUN COMMAND (ONE-LINER) ---

  fcn_prepare -run ; \
  self_mail fcn_GFC_client_started ; \
  fcn_status -wait ; \
  self_mail fcn_client_finished ; \
  fcn_prepare -run_fdr ; \
  fcn_status -wait ; \
  self_mail fcn_client_FDR_finished ; \
  fcn_scripts/fcn_check_runs.tcl ; \
  fcn_promote -no_check -tag GFCA0_FCN_rtl26ww12b_client_refresh ; \
  fcn_mail_summary ; \
  fcn_promote -no_check -run_fev ; \
  fcn_status -wait ; \
  self_mail fcn_client_FEV_finish

  Then after review: $ward/fcn_commit.csh

================================================================================
  KEY INDICATORS — WHAT TO LOOK AT
================================================================================

  1. fdr_cnt (FDR = Formal Design Rule)
  --------------------------------------
  Compares TST vs REF FDR exception counts per partition.

  What to check:
    - fdr_exceptions: Should be CLOSE between TST and REF
    - Small delta (1-2) is normal (new clock/IO tag changes)
    - Large delta = something changed in synthesis -> investigate
    - arrays_mcp: Should be 0 (multi-cycle path arrays)
    - fdr_exceptions_log_errors: Should match TST vs REF

  In WW14_4: All partitions show TST ~2 fewer FDRs than REF.
  This is because new clock tag resolved some ssh_ijtag/ijtag_tck paths.

  2. seq_and_cell (Sequential & Cell counts)
  -------------------------------------------
  Compares Cadence Genus vs Synopsys FC synthesis results.

  What to check:
    - cdns_cell / cdns_seq: Cadence cell and sequential counts
    - snps_cell / snps_seq: Synopsys cell and sequential counts
    - promote: Which tool's netlist gets promoted (always "cdns" here)
    - Large delta between TST/REF = RTL change or synthesis option change

  In WW14_4: par_meu shows biggest change:
    TST: 1,773,284 cells vs REF: 1,827,494 (-54K cells, -3%)
    This is significant — worth investigating.

  3. hips_cnt (HIP = Hard IP instances)
  ---------------------------------------
  What to check:
    - hip_count should match rtl_hip_cnt
    - xml_only=0, rtl_only=0 (no mismatches)
    - par_msid has rtl_only=1 — known waiver

  Check hips_paths: fcn_report_csv.tcl
  EBB pins reference:
    /nfs/site/proj/gfc/gfc.gk.workarea.44/core/integrate_bundle42032/
      target/core/RTP/client_n2/par_msid/par_msid/syn/n2/original_ebbs_pins

================================================================================
  HOW THIS CONNECTS TO YOUR SIO WORK
================================================================================

  1. FCN produces netlists -> Gil uses them for daily builds (FCT)
  2. When FCN releases a new model, a new FCT daily build follows
  3. FDR changes in FCN can cause new timing paths in FCT
  4. Cell count changes (seq_and_cell) affect timing closure effort
  5. The clock tag change (ijtag_cdc_meu_unstamp) may affect
     your DFX timing paths (ssh_ijtag/ijtag_tck domain)

  As SIO, you care about:
    - Did cell counts change significantly? (impacts timing)
    - Are there new FDR exceptions? (new unconstrained paths)
    - Did HIPs change? (impacts physical timing)
    - Clock tag changes -> check DFX and ijtag paths in next daily
