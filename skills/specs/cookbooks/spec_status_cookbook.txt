1. OVERVIEW — What is spec_status?
================================================================================

  Spec status checks whether every cross-partition signal has a proper
  IO delay specification (budget) and whether those budgets are balanced.

  In a partitioned design, each cross-partition signal needs:
    - A "spec" (IO delay budget) on BOTH sides (driver + receiver partition)
    - The two specs must sum to less than the clock period (balanced)
    - No signal should be unconstrained at the partition boundary

  The spec_status directory has 3 sub-directories:

  ┌──────────────────────┬───────────────────────────────────────────────┐
  │ missing_spec/        │ Signals with NO spec at all                  │
  │                      │ → Need to add IO delay constraints           │
  ├──────────────────────┼───────────────────────────────────────────────┤
  │ high_io_delay/       │ Signals with EXCESSIVE IO delay spec         │
  │                      │ → Spec is too generous, eating clock budget  │
  ├──────────────────────┼───────────────────────────────────────────────┤
  │ unbalanced_spec_dir/ │ Signals where driver+receiver specs don't    │
  │                      │ balance to the clock period                  │
  │                      │ → Budget is wasted or impossible to close    │
  └──────────────────────┴───────────────────────────────────────────────┘

2. MISSING SPEC — missing_spec/
================================================================================

  2.1  SUMMARY FILES (start here)
  ─────────────────────────────────
  core_client.missing.spec.0bit.rpt     ← Main summary (CSV format)
  core_client.missing.spec.4bit.rpt     ← Grouped by 4-bit granularity
  core_client.missing.spec.8bit.rpt     ← Grouped by 8-bit
  core_client.missing.spec.16bit.rpt    ← Grouped by 16-bit
  core_client.missing.spec.32bit.rpt    ← Grouped by 32-bit
  core_client.missing.logic_delay_spec.rpt  ← Missing logic delay specs

  Format (CSV):
    par,total_0,missing_0
    par_exe,535,41         ← 41 out of 535 pins missing specs
    par_fe,706,0           ← par_fe is clean — zero missing
    par_msid,697,0         ← par_msid is clean too

  The bit-width grouping collapses bus signals:
    0bit  = each pin counted individually
    4bit  = pins grouped into 4-bit chunks
    32bit = pins grouped into 32-bit chunks
  → Higher grouping shows fewer "missing" because partial buses count as covered

  2.2  PER-PARTITION DETAIL FILES
  ─────────────────────────────────
  <partition>_missing_0bit.rpt          ← List of missing signal names
  <partition>_missing_4bit.rpt          ← Same, 4-bit grouped

  Format (missing report):
    512 icore0/dc2mlwrdatam402h_*_*_*_*_*_[*]     ← 512 pins missing
     32 icore0/dcreqbyteenm402h[*]                 ← 32 pins missing
      1 icore0/update_in                           ← 1 pin missing

  Format (filter file):
    Regex patterns of signals excluded from the "missing" count.
    These are waived/known-OK signals (DFX, scan, test modes, etc.)

  2.3  PIN LIST FILES
  ─────────────────────
  all_pin_no_FT.txt                     ← ALL pins that are NOT feedthrough
  all_<partition>_pin_no_FT.txt         ← Per-partition non-FT pins

  all_splited_ports                     ← Ports that were split

  Format (pin_no_FT):
    icore0/par_pmh/signal_name,, {70.0 mclk_pmh userSpec {}} ,
    ← pin_name, empty, {delay_value clock_domain spec_type {exceptions}}

  Key: "no_FT" = non-feedthrough = real functional pins that need specs
       Feedthrough pins just pass signals through (don't need their own spec)

  2.4  LOGIC DELAY FILES

  all_missing_logic_delay_spec.rpt      ← Full report (usually empty = good)

3. HIGH IO DELAY — high_io_delay/
================================================================================

  3.1  WHAT IT MEANS
  ────────────────────
  Lists signals where the IO delay spec (budget) is HIGHER than expected.
  High IO delay = the partition boundary timing budget is too generous,
  which means less budget remains for logic inside the partition.

  Value shown is the IO delay in picoseconds (ps).

  3.2  REPORT FILES
  ────────────────────
  High_IO_delay_<partition>_report      ← Full list, one pin per line
  full_report_<partition>               ← Full report with all pins
  full_report_<partition>_compressed    ← Compressed (bus bits collapsed)
  full_report_<partition>.ind           ← Index/histogram summary (CSV)
  <partition>_exceptions                ← Signals with timing exceptions

  high_IO_delay_report_all              ← Combined all-partition report

  3.3  FORMAT
  ─────────────
  Full/compressed report:
    tckmode_ip4 24546.0
    ooo_int_lcp_dfx_lcp_fuse_out[*] 15948.0
    bpqantvectorm103h_*_[*] 156.0
    ← signal_name  IO_delay_value(ps)

  Sorted by IO delay (highest first = worst offenders).
  Compressed version collapses bus bits with [*] wildcards.

  Index file (.ind) — histogram of IO delay distribution:
    ,Lines,Lines Compress,-30--10,-10-10,10-30,...,largest_exception
    par_msid,9581,703,1,1,32,97,...,3

    Columns: total lines, compressed lines, then counts per delay bucket
    Delay buckets: -30 to -10, -10 to 10, 10 to 30, ... (ps ranges)

  Exception file:
    ## -I-   count of endpoints of signal in domain clock with exceptions: N
    ← Shows which signals have timing exceptions applied

  3.4  io_check_reports/ SUBDIRECTORY
  ─────────────────────────────────────
  Per-partition sub-reports for specific IO issues:

    <par>_Unconstrained.rpt     ← Pins with NO constraint at all
    <par>_low_budget.rpt        ← Pins with very tight (low) budget
    <par>_small_FT.rpt          ← Feedthrough pins with small spec
    <par>_no_launch_clock.rpt   ← Pins missing launch clock definition
    <par>_no_capture_clock.rpt  ← Pins missing capture clock definition
    <par>_logic_delay_with_default_FT_spec.rpt  ← Logic delay using default
    <par>_total.rpt             ← All pins flagged for this partition
    <par>_filter                ← Filter patterns (waived signals)

  3.5  QUICK CHECK COMMANDS
  ──────────────────────────
  # Top IO delay offenders for a partition (compressed):
    head -20 high_io_delay/full_report_par_msid_compressed

  # Histogram summary:
    cat high_io_delay/full_report_par_msid.ind

  # Unconstrained partition pins:
    cat high_io_delay/io_check_reports/par_msid_Unconstrained.rpt

4. UNBALANCED SPEC — unbalanced_spec_dir/
================================================================================

  4.1  WHAT IT MEANS
  ────────────────────
  An "unbalanced" spec means the IO delay budgets on the two sides of a
  cross-partition signal DON'T properly sum to the available clock margin.

  The balance formula:
    margin = period - uncertainty + skew - FT_delay - spec_driver - spec_receiver

  If margin is NEGATIVE → the specs are TOO LARGE (overbudgeted)
  If margin is very POSITIVE → budget is wasted (underutilized)

  4.2  REPORT FORMAT
  ────────────────────
  Unbalanced_spec_report                    ← Full report (all signals)
  Unbalanced_spec_report_compress           ← Compressed (bus collapsed)
  Unbalanced_spec_report_compress_<par>     ← Per-partition compressed
  subfc_Unbalanced_spec_report              ← Sub-FC level report
  Unbalanced_dfx_spec_report                ← DFX signals only
  Historam.csv                              ← Histogram of balance margins

  Each line format (pipe-delimited):
    margin | driver_pin | receiver_pin | constraint_info | drv_spec | rcv_spec | FT_info | FT_delay | balance_equation |

  Example:
    -46.00 | icore0/par_msid/dsftisvalidbrm185h | icore0/par_fe/dsftisvalidbrm185h | constrained mclk_msid mclk_fe | 145.00 userSpec | 67.00 userSpec |  | 0.00 | 186.00 - 0.00 + -20.00 - 0.00  - 67.00 - 145.00  |

  Reading this:
    margin    = -46.00 ps (NEGATIVE = overbudgeted by 46 ps!)
    driver    = par_msid/dsftisvalidbrm185h
    receiver  = par_fe/dsftisvalidbrm185h
    drv_spec  = 145.00 ps (userSpec = manually set)
    rcv_spec  = 67.00 ps (userSpec)
    FT_delay  = 0.00 ps (no feedthrough)
    equation  = 186.00 - 0.00 + -20.00 - 0.00 - 67.00 - 145.00 = -46.00
              = period - uncertainty + skew - FT - rcv_spec - drv_spec

  4.3  FEEDTHROUGH SIGNALS
  ──────────────────────────
  Some signals pass through a 3rd partition (feedthrough):
    :icore0/par_msid/ROAryBobidM202H_2___FEEDTHRU_0(40.26)
    ← Feedthrough through par_msid, adding 40.26 ps delay

  The FT delay is SUBTRACTED from the available margin, making balance harder.

  4.4  HISTOGRAM (Historam.csv)
  ──────────────────────────────
  Shows distribution of balance margins per partition:
    ,-100,-75,-50,-35,-20,20,35,50,75,100,
    par_msid,0,0,0,1,2,2,0,0,0,0,

  Read as: par_msid has
    1 signal in the -50 to -35 range (dangerous)
    2 signals in the -35 to -20 range (tight)
    2 signals in the 20 to 35 range (healthy)

  Negative = overbudgeted (driver+receiver specs exceed period)
  Positive = underutilized (budget left on the table)

  4.5  QUICK CHECK COMMANDS
  ──────────────────────────
  # Histogram overview:
    cat unbalanced_spec_dir/Historam.csv

  # Worst unbalanced signals for a partition:
    head -10 unbalanced_spec_dir/Unbalanced_spec_report_compress_par_msid

  # Search for a specific signal:
    grep "signal_name" unbalanced_spec_dir/Unbalanced_spec_report

5. KEY CONCEPTS — How specs work in partitioned design
================================================================================

  5.1  THE SPEC BUDGET MODEL
  ────────────────────────────
  Clock period = 186 ps (for mclk at this frequency)
  Clock uncertainty = ~20 ps
  Available budget = period + skew - uncertainty = ~166 ps

  This budget is SPLIT between:
    - Driver partition IO delay spec (how much time to get to boundary)
    - Receiver partition IO delay spec (how much time from boundary to FF)
    - Feedthrough delay (if signal passes through a 3rd partition)

  Example:
    drv_spec = 145 ps + rcv_spec = 67 ps = 212 ps
    Available = 166 ps
    Margin = 166 - 212 = -46 ps → OVERBUDGETED!

  5.2  SPEC TYPES
  ─────────────────
  userSpec     — manually set by the TD team (most common)
  defaultSpec  — auto-generated default value
  FT spec      — feedthrough specification

  5.3  FEEDTHROUGH (FT)
  ───────────────────────
  When a signal crosses from partition A → partition C but physically
  routes THROUGH partition B, partition B has a "feedthrough" pin.

  FT pins appear as: SignalName__FEEDTHRU_0, SignalName__FEEDTHRU_1
  (two pins: one entry, one exit from the feedthrough partition)

  The feedthrough delay consumes part of the available budget,
  making it harder to meet timing for the real driver/receiver.

  5.4  PARTITIONS IN THIS DESIGN
  ─────────────────────────────────
  14 partitions in core_client:
    par_msid, par_fe, par_meu, par_ooo_int, par_ooo_vec,
    par_exe, par_pmh, par_pm, par_mlc, par_fmav0, par_fmav1,
    par_tmul, icore (top-level), dfx

6. WORKFLOW — How to use these reports
================================================================================

  6.1  DAILY CHECK (quick health)
  ─────────────────────────────────
  1. Check missing specs:
       cat missing_spec/core_client.missing.spec.0bit.rpt
     → par_fe=0, par_msid=0 is good
     → par_tmul=151/151 is 100% missing (known — it's a stub)

  2. Check unbalanced histogram:
       cat unbalanced_spec_dir/Historam.csv
     → Look for counts in the -100 to -35 columns (dangerous)

  3. Check worst unbalanced:
       head -5 unbalanced_spec_dir/Unbalanced_spec_report_compress_par_msid
     → Margin < -20 ps needs attention

  6.2  INVESTIGATING A FAILING SIGNAL
  ─────────────────────────────────────
  If a cross-partition path is failing:
  1. Check if it has a spec:
       grep "signal_name" missing_spec/all_pin_no_FT.txt
     → Shows the current spec value and clock domain

  2. Check if specs are balanced:
       grep "signal_name" unbalanced_spec_dir/Unbalanced_spec_report
     → Shows margin, driver spec, receiver spec, feedthrough

  3. Check IO delay value:
       grep "signal_name" high_io_delay/full_report_par_msid
     → Shows the current IO delay

  4. Check if it's a feedthrough:
       grep "signal_name" missing_spec/all_pin_with_FEEDTHRU.txt
       grep "signal_name" missing_spec/pure_ft_pin.txt

  6.3  REQUESTING SPEC CHANGES
  ──────────────────────────────
  If a spec needs adjustment (rebalancing):
  - TD team owns the spec values (userSpec)
  - File a request to change drv_spec or rcv_spec
  - The goal: margin should be slightly positive (0 to +10 ps ideal)
  - Moving budget: increase one side = decrease the other side
