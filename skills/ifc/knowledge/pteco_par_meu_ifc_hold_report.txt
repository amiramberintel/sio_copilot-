SUMMARY
-------
521 functional IFC hold violations in par_meu across 5 hold/min corners.
36 unique signal groups. 190 additional DFT/JTAG paths (ignored below).

Worst: -51ps (SpecIsValid512ForLDBundleCancelM801H -> VecLdPortWriteMuxCtlM802H)
Most of the violations are on 3 interfaces:
  - par_exe -> par_meu  (vector load cancel, 132 paths, worst -51ps)
  - par_pmh -> par_meu  (DCU L1 data/tag, 337 paths, worst -13ps)
  - par_meu -> par_pmh  (snoop/GLB array, 12 paths, worst -6ps)


DETAILED IFC HOLD VIOLATIONS
=============================
Grouped by interface, sorted worst-first within each group.
Paths shown are per icore (x2 for icore0+icore1).

===============================================================================
INTERFACE 1: par_exe -> par_meu  (mclk_exe -> mclk_meu)
  132 paths | Worst: -51ps | ALL 3 hold corners
===============================================================================

This is the WORST IFC hold interface. All paths are vector load cancel/PRF
operations where EXE sends speculative cancel signals to MEU.

  #  Slack  Signal (SP -> EP)                                        #paths  Corners
  -  -----  -------------------------------------------------------  ------  -------
  1  -51ps  SpecIsValid512ForLDBundleCancelM801H                       30    all 3
            -> VecLdPortWriteMuxCtlM802H
            WHAT: 512-bit vector load bundle cancel valid signal.
                  EXE decides if a speculative 512-bit vector load
                  should be cancelled. MEU captures the cancel to
                  update the port write mux control.
            WHY HOLD FAIL: Short data path (M801->M802 = 1 stage)
                  + clock skew mclk_exe arrives early vs mclk_meu.

  2  -47ps  SpecPDstForLDBundleCancelM801H                            18    all 3 + func.min_nom
            -> VecLdPortWriteMuxCtlM802H
            WHAT: Physical destination for the cancelled load bundle.
                  Tells MEU which PRF destination to cancel.

  3  -30ps  SpecIsValid512ForLDBundleCancelM801H                       10    fast, fast_cold
            -> RSVecLdPRFWrCancelM802H
            WHAT: Same cancel valid, different endpoint — cancels the
                  PRF write for the vector load.

  4  -29ps  SpecPDstForLDBundleCancelM801H                            10    all 3
            -> RSVecLdPRFWrCancelM802H
            WHAT: Physical destination for PRF write cancel.

  5  -28ps  PtPwrUpM303H                                              36    all 3
            -> RSVecUopDepOnLoadM303H
            WHAT: Power-up signal from EXE at stage M303H tells MEU
                  RS that the vector uop depends on a load. Same-stage
                  transfer (303->303) — very tight hold timing.

  6  -25ps  PtPwrUpM302H                                              16    all 3
            -> RSVecUopDepOnLoadM303H
            WHAT: Earlier power-up (M302) to same endpoint.

  7  -22ps  BypEnL0TmpM302H                                           12    all 3
            -> RSVecUopDepOnLoadM303H
            WHAT: Bypass enable from L0 temp at M302H feeds into
                  the load dependency check at M303H.

  ROOT CAUSE: The exe->meu clock skew on hold corners causes mclk_exe
  to fire early relative to mclk_meu. Combined with short data paths
  (1-stage transfers), the data arrives too early and violates hold.
  
  FIX OPTIONS:
  a) Add delay cells on the cross-partition data path (PTECO should do)
  b) Adjust IFC hold spec to add margin
  c) Clock skew adjustment between mclk_exe and mclk_meu
