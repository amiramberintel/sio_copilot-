  The MB2 cell is placed at (1160, 783) because RORtLdIncM904H[4]

================================================================================
  KEY LESSONS FROM THIS ANALYSIS
================================================================================

  1. ALWAYS CHECK INPUT PATHS BEFORE PROPOSING FF MOVEMENT
     The D2 input path had only +38.6ps margin. Moving the FF to the
     "obvious" location (850,600) would have created a new -16ps violation.
     This reduced our de-merge estimate from +65-90ps to +20-30ps.

  2. MBIT MERGE IS A COMMON ROOT CAUSE
     Synthesis packs FFs by clock domain, ignoring output direction.
     When two bits in an MB2 go to different partitions, one will suffer.

  3. ONE TIP CAN FIX MULTIPLE ENDPOINTS
     All 6 failing paths shared the same par_ooo_int buffer chain.
     One TIP plan fixes all 6 simultaneously.

  4. CHECK EXISTING TIP PLANS ON SAME BOUNDARY
     specstall already had a TIP on par_ooo_int → par_ooo_vec.
     Used it as reference for route geometry, metal layers, and repeaters.

  5. SPEC ANALYSIS REVEALS TRUE ROOT CAUSE
     par_ooo_int: spec=34ps, used=140.5ps → the REAL problem
     par_ooo_vec: spec=134ps, used=115.2ps → already within budget
     Without spec check, you might waste time optimizing the wrong side.

  6. WIRE-DOMINATED PATHS NEED TIP, NOT JUST ECOs
     Wire = 50.8% of total delay. Upsizing gates helps on cell-dominated
     paths, but for wire-dominated paths you need controlled routing (TIP).

  7. PLACEMENT TRADEOFF TABLES QUANTIFY RISK
     Don't just say "move the FF closer." Build a table showing output
     gain vs input degradation at 4-5 positions to find the safe range.

  8. USE bbox {50 50} + move_objects FOR TIP FILES
     Cookbook Section 14 pattern: get_cells + bbox_percentage {50 50}
     + move_objects. Simpler and more reliable than get_bounds/4-node pattern.
