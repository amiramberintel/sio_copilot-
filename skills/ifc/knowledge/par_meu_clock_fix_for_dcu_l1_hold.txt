 1.  THE PROBLEM — CLOCK SKEW KILLS HOLD
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  PATH:  par_pmh/DcL1RdDataIntM408H (mclk_pmh) → par_meu/WSBDataArrayMnnnH (mclk_meu)
  
  3,149 hold-violating paths at min_low  |  WNS = -111ps
  3,362 setup-violating paths at max_high |  WNS = -52ps
  
  The mclk_meu capture clock to the WSBDataArray/SnpBufDataArray registers
  has the HIGHEST latency of any par_meu endpoint group:
  
    333ps at min_low  (next worst = 313ps for rseuimmdataqb)
    → This is 108ps ABOVE the mclk_meu average of 225ps


  ┌────────────────────────────────────────────────────────────────────────┐
  │  CLOCK SKEW AT DCU L1 DATABANK:                                       │
  │                                                                        │
  │                          min_low (hold)     max_high (setup)           │
  │  ─────────────────────   ──────────────     ────────────────           │
  │  Capture (mclk_meu):        333ps               73ps                  │
  │  Launch  (mclk_pmh):        142ps               91ps                  │
  │  Clock Skew:                +191ps              -18ps                  │
  │                                                                        │
  │  At min_low: capture arrives 191ps AFTER launch → HOLD FAILS          │
  │  At max_high: capture arrives 18ps BEFORE launch → SETUP TIGHT        │
  │                                                                        │
  │  HOLD needs: less skew (reduce capture or increase launch)             │
  │  SETUP needs: same or more skew (capture late helps setup)             │
  │  → These requirements CONFLICT — any clock change trades one for other │
  └────────────────────────────────────────────────────────────────────────┘

 2.  VOLTAGE SCALING — WHY min_low IS SO MUCH WORSE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  The capture clock latency scales 4.56x from max_high to min_low,
  but launch only scales 1.56x:

  ┌────────────────────────┬──────────┬──────────┬──────────┐
  │ Clock path             │ max_high │ min_low  │ Scaling  │
  ├────────────────────────┼──────────┼──────────┼──────────┤
  │ DCU L1 capture (meu)   │   73ps   │  333ps   │  4.56x   │ ← PROBLEM
  │ DCU L1 launch  (pmh)   │   91ps   │  142ps   │  1.56x   │
  │ mclk_meu average       │   96ps   │  225ps   │  2.34x   │
  │ mclk_pmh average       │   90ps   │  212ps   │  2.36x   │
  └────────────────────────┴──────────┴──────────┴──────────┘

  WHY 4.56x? The capture clock path has:
  - 3 ICGs (each with enable timing sensitive to voltage)
  - 12+ CTS buffer stages (each adds ~15-25ps at min_low vs ~5-7ps at max_high)
  - ICGs scale ~3-4x, CTS buffers scale ~2.5-3.5x
  - The compounding of 15+ stages creates the extreme 4.56x ratio

  The launch clock has only 2 ICGs + 6 CTS stages → scales moderately (1.56x)

  IMPLICATION:
  Any clock fix is amplified differently at different corners:
  - Removing 1 CTS buffer stage from capture saves ~5ps at max_high
    but saves ~15-20ps at min_low (3-4x leverage!)
  - This asymmetry can be USED to our advantage

 5.  SETUP IMPACT ANALYSIS — WILL WE BREAK SETUP?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Current setup status at max_high: WNS = -52ps, 3,362 failing paths
  Setup is ALREADY failing — but with different characteristics:
  
  ┌────────────────────────────────────────────────────────────────────────┐
  │  SETUP PATH CHARACTERISTICS (max_high):                               │
  │                                                                        │
  │  Clock skew = -18ps (capture 18ps BEFORE launch)                      │
  │  This means capture clock is FASTER than launch at max_high            │
  │  → Opposite of the hold problem!                                      │
  │                                                                        │
  │  Setup equation:                                                       │
  │    data_required = capture_edge + capture_latency - setup_time         │
  │    data_arrival  = launch_edge  + launch_latency  + data_delay         │
  │    setup_slack   = data_required - data_arrival                        │
  │                                                                        │
  │  If we REDUCE capture latency by Δ:                                   │
  │    data_required decreases by Δ → setup slack DECREASES by Δ          │
  │    BUT at max_high Δ is SMALL (1/4.56 of the min_low change)          │
  └────────────────────────────────────────────────────────────────────────┘

  THE KEY LEVERAGE: 4.56x Voltage Scaling Ratio
  
  ┌────────────────────────────────────────────────────────────────────────┐
  │                                                                        │
  │  If par_meu CTS removes 1 buffer stage from capture path:            │
  │                                                                        │
  │    At min_low:  saves  ~20ps  (hold improvement)                      │
  │    At max_high: costs  ~5ps   (setup degradation)                     │
  │                                                                        │
  │  If par_meu bypasses CG_RC ICG:                                       │
  │    At min_low:  saves  ~22ps  (hold improvement)                      │
  │    At max_high: costs  ~7ps   (setup degradation)                     │
  │                                                                        │
  │  Combined (1 buffer + ICG bypass):                                    │
  │    At min_low:  saves  ~42ps  (3,149 → ~1,000 paths remaining)       │
  │    At max_high: costs  ~12ps  (WNS -52 → ~-64ps)                     │
  │                                                                        │
  │  Is -12ps setup degradation acceptable?                               │
  │    Current WNS = -52ps — already needs ECO/buffer insertion            │
  │    -64ps is worse but in same ballpark                                │
  │    PTECO can fix setup violations with buffers (unlike hold!)         │
  │    Hold ECO is MUCH harder than setup ECO → prioritize hold fix       │
  │                                                                        │
  └────────────────────────────────────────────────────────────────────────┘


  SETUP SANITY CHECK — How many setup paths would be created?

  Current at max_high: 3,362 paths at -52ps WNS
  If capture reduced by 12ps: ~3,500 paths at -64ps WNS (estimate)
  
  But these are EXTERNAL paths (par_pmh→par_meu):
  - Setup can be fixed by PTECO with buffers on DATA path (adding delay)
  - Hold CANNOT be fixed by PTECO (already tried, failed due to L/T codes)
  - Setup ECO is a standard PTECO capability
  
  CONCLUSION: Accept setup degradation to gain hold improvement.
  Hold is UNFIXABLE by PTECO (buffer insertion adds delay = helps hold only).
  Setup fixes require cell sizing/swapping (harder, limited by Vt/area) or
  SIO2PO departure bounds on par_pmh to reduce data path delay.
