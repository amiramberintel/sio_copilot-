  - 8 banks × ~150-160 paths/bank = 1230 total paths across 2 cores

 5.  ROOT CAUSE ANALYSIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ┌────────────────────────────────────────────────────────────────────────┐
  │  ROOT CAUSE #1: MASSIVE POSITIVE CLOCK SKEW (+143ps)                  │
  │                                                                        │
  │  The capture clock (mclk_meu) has 3 ICGs vs 2 for launch (mclk_pmh). │
  │  Plus 12 CTS stages vs 6 for launch.                                   │
  │                                                                        │
  │  This means the capture clock arrives 143ps AFTER the launch clock.    │
  │  For hold timing: data must be STABLE for 143ps after launch.          │
  │  But the data path is only 97ps long → data changes too fast.          │
  │                                                                        │
  │  WHY 3 ICGs on capture?                                                │
  │    ICG1: ClkL1IslandMH  — L1 island power gating (architectural)      │
  │    ICG2: ClkWSBWr4094MH — WSB write bank gating (power management)    │
  │    ICG3: CG_RC_CG_HIER  — Register control gate (auto-inserted)       │
  │                                                                        │
  │  The 3rd ICG (CG_RC) is auto-inserted by CTS for clock gating.        │
  │  This is the most likely candidate for optimization.                   │
  └────────────────────────────────────────────────────────────────────────┘
  
  ┌────────────────────────────────────────────────────────────────────────┐
  │  ROOT CAUSE #2: CROSS-PARTITION — NO CTS COORDINATION                 │
  │                                                                        │
  │  mclk_pmh (par_pmh) and mclk_meu (par_meu) are balanced              │
  │  independently during CTS.                                             │
  │                                                                        │
  │  Each partition CTS only sees its own clock tree.                       │
  │  Neither knows about the cross-partition hold requirement.             │
  │  The 143ps gap is an artifact of independent clock tree synthesis.     │
  └────────────────────────────────────────────────────────────────────────┘
  
  ┌────────────────────────────────────────────────────────────────────────┐
  │  ROOT CAUSE #3: ECC LOGIC IS TOO FAST                                  │
  │                                                                        │
  │  The data path goes through dcl1_dat_hmatrix (ECC syndrome XOR+MUX)   │
  │  which is only 6 cells deep (~97ps total including reg-to-reg).        │
  │                                                                        │
  │  For a cross-partition path with 143ps skew, you'd need ~170ps         │
  │  of data delay (143 + 11 hold_time + 18 uncertainty - margin).         │
  │  The data path is 73ps SHORT of what's needed.                         │
  └────────────────────────────────────────────────────────────────────────┘
  
  ┌────────────────────────────────────────────────────────────────────────┐
  │  ROOT CAUSE #4: LIBRARY CELL DENSITY (L code in PTECO)                 │
  │                                                                        │
  │  PTECO tried to insert hold buffers but failed with code "L" (area)    │
  │  and "T" (timing too tight).                                           │
  │                                                                        │
  │  The MBIT register banks are extremely dense — no room for buffers.    │
  │  Physical-aware ECO cannot find open sites near the endpoints.         │
  └────────────────────────────────────────────────────────────────────────┘

 7.  WHAT ABOUT CLOCK FIX? (user_useful_skew.tcl approach)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  The user_useful_skew.tcl approach (seen in par_ooo_vec for similar paths)
  is the STANDARD mechanism in this design for cross-partition hold fixes.
  
  EXAMPLE (from par_ooo_vec PO for miintwbrsderrm807h):
    user_useful_skew.tcl:  set_clock_latency -offset +10ps <capture_flop>
    user_reg_bound2port.tcl: sio2po_magnet on the port
    → This added 10ps to capture register, fixing hold paths
  
  FOR DCU L1 DATABANK, equivalent approach:
  
  IN par_meu user_useful_skew.tcl:
    # Pull WSBDataArray capture clock earlier by 5-10ps
    set_clock_latency -offset -5 [get_pins \
      */dcibbankd*/auto_vector_MBIT_WSBDataArrayMnnnH_reg_*/CP]
  
  IN par_pmh user_useful_skew.tcl:
    # Push DcL1RdData launch clock later by 10-15ps
    set_clock_latency -offset +10 [get_pins \
      */dcl1data/auto_vector_MBIT_DcL1RdDataIntM408H_reg_*/CP]
  
  COMBINED: ~15-25ps improvement across the interface
  
  ⚠️  CRITICAL: The 184 paths that fail BOTH hold AND setup
     CANNOT be fixed by clock skew alone.
     These need DATA PATH restructure (RTL change or buffer insertion
     in a less-dense area).

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ──────────────────             ───────────────────
  glbdrv  (29ps)                 glbdrv  (48ps)         +19ps gap from start
