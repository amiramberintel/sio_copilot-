  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 0.6: CHECK CELL TYPES — ULVT UPGRADE CANDIDATES                  │
  └─────────────────────────────────────────────────────────────────────────┘

  From the timing report, look at EVERY cell name on the data path.
  Cell name suffix tells you the Vt type:

    CPDLVT   → LVT  (Low Vt)       — CAN be upgraded to ULVT for ~5-15ps gain
    CPDULVT  → ULVT (Ultra-Low Vt)  — already fastest, no upgrade possible
    CPDHVT   → HVT  (High Vt)       — upgrade to LVT or ULVT for big gain!
    CPDSVT   → SVT  (Standard Vt)   — upgrade to LVT for moderate gain

  WHAT TO CHECK:
    □ Count LVT cells on the path — each is a ULVT upgrade candidate
    □ Count HVT/SVT cells — these are LOW-HANGING FRUIT (biggest gain per swap)
    □ Focus on cells with LARGE incremental delay (>10ps)
    □ Buffers and drivers benefit most from Vt upgrade

  HOW TO CHECK with pt_client:
    The cell type is visible in the report_timing output — look at the cell
    name in parentheses at each stage, e.g.:
      (BUFFD2BWP156HPNPN3P48CPDLVT)     ← LVT buffer, can upgrade to ULVT
      (BUFFSKFD14BWP156HPNPN3P48CPDULVT) ← already ULVT, no upgrade
      (BUFFDHDVGLLKGD18BWP156HPNPN3P48CPDULVT) ← already ULVT

  EXPECTED GAIN PER CELL:
    HVT  → ULVT:  ~10-20ps per cell (big win!)
    SVT  → ULVT:  ~8-15ps per cell
    LVT  → ULVT:  ~3-8ps per cell
    Power cost: ULVT uses more leakage power, but on critical paths it's worth it

  REAL EXAMPLE (RSMOClearVM803H):
    Path has mix of CPDLVT and CPDULVT cells:
      invs_route_opt_.../Z (BUFFD2BWP156HPNPN3P48CPDLVT) ← LVT, upgrade candidate
      invs_place_FE_.../Z  (BUFFSKRD4BWP156HPNPN3P48CPDLVT) ← LVT, upgrade candidate
    → Upgrading 2 LVT buffers to ULVT could gain ~6-16ps
    → This alone won't close -71.6ps but helps as part of combined fix

  NOTE: pt_client is READ-ONLY — you cannot run size_cell to swap Vt.
    To request the swap: create a TIP file or ask SIO for cell swaps.

  ⚠️ CRITICAL: Before proposing any cell swap or trusting PT timing, check

  Illegal cells list:

  ~15,746 banned cells. Main categories:
    CPDELVT  → BANNED — elvt/svt cells not allowed
    CPDLVTLL → BANNED — lvtll cells not allowed
    PNNP OD  → BANNED — only PNPN OD flavor allowed
    M2 pitch → BANNED — illegal default M2 pitch
    Non-ULVT clock cells → BANNED — only ULVT for clock cells

  Quick cross-check from timing report:
    # Extract cell ref_names from PT report
    grep -oP '\w+BWP156H\w+' /tmp/path_report.txt | sort -u > /tmp/path_cells.txt

    # Check against illegal list
    while read cell; do
      grep -q "$cell" $wa/project/GFCN2CLIENTA0/illegal.txt && \
        echo "⚠️  ILLEGAL: $cell"
    done < /tmp/path_cells.txt

  If illegal cell found on path:
    → The reported slack is OPTIMISTIC (real slack is worse)
    → Replace with legal alternative and recalculate
    → Common: ELVT→LVT loses ~5-10ps, so add that to the gap

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 0.7: CHECK FF LOCATIONS — DOES PLACEMENT MAKE SENSE?             │
  └─────────────────────────────────────────────────────────────────────────┘

  From the timing report, extract Source FF and Endpoint FF coordinates.
  Look at the (X,Y) location in the Location column of the report.

  WHAT TO CHECK:
    □ Source FF (X,Y) — is it near the partition port?
    □ Endpoint FF (X,Y) — is it near the partition port?
    □ Port location (X,Y) — is it between the two FFs?
    □ Manhattan distance: |X_sp - X_port| + |Y_sp - Y_port| for source side
    □ Manhattan distance: |X_port - X_ep| + |Y_port - Y_ep| for dest side
    □ Total path distance: source FF → port → endpoint FF

  RED FLAGS — placement does NOT make sense:
    ⚠ FF is >500μm from port → long wire, big delay, PLACEMENT ISSUE
    ⚠ Source FF and endpoint FF on SAME side of port → detour path
    ⚠ Port is NOT between the two FFs → port location issue (Technique 3)
    ⚠ One FF is far but the other is close → move the far FF (Technique 9/TIP)

  HOW TO CHECK with pt_client:
    Locations are in the -physical column of report_timing output.
    The (X,Y) coordinates are in DBU (database units). In GFC N2P: 1 DBU = 1nm.

    ┌───────────────────────────────────────────────────────────────────────┐
    │  COORDINATE CONVERSION: DBU → MICRONS (μm)                          │
    │                                                                      │
    │  Formula:  X_μm = X_DBU / 1000                                      │
    │                                                                      │
    │  Example:                                                            │
    │    report_timing shows:  Location = (1363464, 93574)                 │
    │    In microns:  (1363464/1000, 93574/1000) = (1363.5, 93.6)μm       │
    │                                                                      │
    │  Manhattan distance:                                                 │
    │    dist_μm = (|X1-X2| + |Y1-Y2|) / 1000                            │
    └───────────────────────────────────────────────────────────────────────┘

    Quick distance calculation:
      Distance (μm) = (|X1-X2| + |Y1-Y2|) / 1000

  WHAT MAKES SENSE vs WHAT DOESN'T:
    ✓ GOOD: SP at (700,500), Port at (652,505), EP at (440,357)
      → SP near port, port between FFs, path flows naturally
    ✗ BAD:  SP at (1160,783), Port at (652,505), EP at (440,357)
      → SP is 580μm from port! Too far, needs placement fix

  REAL EXAMPLE (RSMOClearVM803H):
    Source FF:  (1160, 783) in par_ooo_int/rsrptint
    Port:       (652, 505)  abutted boundary
    Endpoint FF: (440, 357) in par_ooo_vec/rob
    
    Source → Port:  |1160-652| + |783-505| = 508 + 278 = 786μm  ← TOO FAR!
    Port → EP:      |652-440| + |505-357| = 212 + 148 = 360μm   ← acceptable
    Total:          786 + 360 = 1146μm total path distance

    VERDICT: Source FF placement does NOT make sense!
      → FF at (1160,783) is 786μm from port — this is the main problem
      → Fix: TIP bound to move FF closer to port, target (1000-1100, 700-780)
      → But D2 input has only +38.6ps margin — limits how close we can move

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 0.8: CHECK SPEC BUDGET IMPACT — BEFORE MOVING ANY FF             │
  └─────────────────────────────────────────────────────────────────────────┘

  BEFORE proposing to move a FF, check the spec budget using spec_status
  reports. Moving a FF changes the clock latency on that FF, which changes
  the available timing budget between partitions.

  See full details: cookbooks/spec_status_cookbook.txt

  THE BUDGET FORMULA:
    margin = period - uncertainty + skew - FT_delay - spec_driver - spec_receiver

  EXAMPLE (mclk at 186ps period):
    period       = 186 ps
    uncertainty  = ~20 ps
    skew         = clock_EP - clock_SP  (changes when FF moves!)
    budget       = 186 - 20 + skew = ~166ps + skew

  WHY FF MOVEMENT AFFECTS THE BUDGET:
    When you move a FF closer to the port:
      ✓ Data path gets SHORTER (less wire delay) — this is what we want
      ✗ Clock latency CHANGES — the FF sees a different clock tree tap
      ✗ Skew between SP and EP changes — shifts the budget balance

    If the new location has WORSE clock latency:
      → The budget gain from shorter wire may be eaten by clock penalty
      → You MUST check this BEFORE requesting the TIP!

  HOW TO CHECK (OFFLINE — from WA on disk):
    Navigate to: <WA>/sta_pt/<corner>/reports/spec_status/

    # 1. Check if signal has a spec (not missing)
    grep "<signal_name>" missing_spec/all_pin_no_FT.txt
    → Shows: {delay_value clock_domain spec_type}

    # 2. Check if budget is balanced
    grep "<signal_name>" unbalanced_spec_dir/unbalanced_spec_summary.csv
    → Shows: margin | driver_pin | receiver_pin | clocks | specs | formula

    # 3. Check if spec is too generous (room to shift)
    grep "<signal_name>" high_io_delay/high_io_delay_summary.csv

  HOW TO CHECK (LIVE — via pt_client):
    # Check clock latency at CURRENT FF location
    pt_client.pl -m latest_gfcn2clienta0_bu_prp_func.max_high.T_85.typical \
      -c 'report_clock_timing -type latency \
        -to icore0/<partition>/<sub>/<FF_name>/CP -nosplit'

    → Record the current clock latency
    → After FF moves, the new clock latency will be different
    → Run the same command on the NEXT daily build to verify

  REAL EXAMPLE (RSMOClearVM803H):
    Current SP FF clock latency at (1160, 783):  86.0ps (from path report)
    Current EP FF clock latency at (440, 357):   68.2ps (from path report)
    Current skew: 68.2 - 86.0 = -17.8ps (EP sees clock EARLIER)

    If we move SP FF to (1050, 750):
      → Clock latency will change (different clock tree tap point)
      → Data path gets ~130μm shorter → saves ~15-25ps wire delay
      → But clock penalty could eat 5-15ps of that gain
      → Net gain estimate: +10-20ps (not the full wire savings!)

    ALWAYS check spec_status AFTER the fix lands to verify budget balance.

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ STEP 0.9: CHECK PORT LOCATION — CAN MOVING THE PORT HELP?             │
  └─────────────────────────────────────────────────────────────────────────┘

  Cross-partition paths go through PORTS at the partition boundary.
  If the port is far from the critical FFs, wire delay is wasted on the
  port-to-FF segment. Moving the port ALONG the boundary might help.

  ⚠️  KEY CONSTRAINT: Ports live ON the partition boundary edge.
     They can only slide UP/DOWN (Y-axis) along the edge — NOT move in X!
     The X-distance from boundary to FF is FIXED by the partition shape.

  HOW TO CHECK (OFFLINE — from WA on disk):
    # Get port location and geometry for crossing signals:
    <WA>/sta_pt/<corner>/reports/port_location_outside_bbox.csv

    # CSV columns:
    # name, abutted, dist_to_bbox, st_point_x, st_point_y,
    # en_point_x, en_point_y, port_point_x, port_point_y,
    # port_point_x_abutted, port_point_y_abutted,
    # fis_count, fos_count, wns, is_moved, td_x, td_y

    grep "<signal_name>" port_location_outside_bbox.csv

  WHAT TO LOOK AT:
    1. port_point (x,y)   — current port location
    2. st_point (x,y)     — worst-path FF location
    3. dist_to_bbox       — if >0, port is OUTSIDE partition bbox (bad!)
    4. fis_count/fos_count — fanin/fanout count (HIGH = risky to move)

  DECISION MATRIX:
    ┌─────────────────────────────────────────────────────────────────┐
    │ Condition                      │ Port Move Helps?              │
    ├─────────────────────────────────────────────────────────────────┤
    │ Big Y gap, small X gap         │ ✅ YES — slide port in Y     │
    │ Big X gap, small Y gap         │ ❌ NO  — X is fixed by edge  │
    │ Big X gap AND big Y gap        │ ⚠️  PARTIAL — Y gain only    │
    │ Port already close to FF       │ ❌ NO  — nothing to gain     │
    │ fos_count > 50                 │ ⚠️  DANGEROUS — many paths   │
    │ dist_to_bbox > 0               │ 🔴 PORT IS OUTSIDE BBOX!     │
    └─────────────────────────────────────────────────────────────────┘

  GAIN ESTIMATION:
    • Every ~100μm saved in wire ≈ 5-8ps delay reduction
    • But ONLY the Y-component can be changed by port sliding
    • Calculate: Y_saved = |port_y - FF_y|
    • If Y_saved < 50μm → negligible gain (<3ps), skip
    • If Y_saved > 200μm → worth investigating (~10-15ps)

  FANOUT CHECK — CRITICAL:
    Before moving ANY port, check its fanout:
    • fos_count < 10  → safe to move (few paths affected)
    • fos_count 10-50  → check carefully (run slack on neighbors)
    • fos_count > 50  → ⚠️  VERY RISKY — moving helps this path
                          but could BREAK many others!

  REAL EXAMPLE (PfxErrVecM105H path):
    Crossing 2 — ifclipm105h_0_[12] (par_fe → par_msid):
      Port:    (1491, 129) — on partition boundary
      EP FF:   (1198, 46)
      X gap:   293μm (FIXED by boundary!)
      Y gap:   83μm  (slideable)
      fis=7, fos=6 (low fanout — safe to move)
      → Moving port from y=129 → y=46 saves only ~83μm in Y
      → Estimated gain: ~3-5ps (NOT worth it for -69ps deficit)
      → The REAL problem: 293μm X-distance is structural

    Crossing 1 — ildpipectlforfem100h_enablestage102to103 (par_msid → par_fe):
      Port:    (1491, 228)
      SP FF:   (1419, 180)
      X gap:   72μm, Y gap: 48μm → total ~87μm
      fis=49, fos=221 ← ⚠️ DO NOT TOUCH — 221 fanout!
      → Already close, no gain, and too risky

  WHEN PORT MOVE IS THE RIGHT FIX:
    ✅ Port is far in Y from the critical FF (>200μm Y gap)
    ✅ Low fanout (fos < 10) — won't break other paths
    ✅ X gap is small — most of the wire is in Y direction
    ✅ Combined with TIP (move FF closer to boundary in X)

  WHEN PORT MOVE IS NOT THE RIGHT FIX:
    ❌ Big X gap (FF far from boundary) — port can't fix this
    ❌ High fanout (fos > 50) — too many paths affected
    ❌ Port already close to FF — nothing to gain
    ❌ Whole bus failing — systemic issue, not port placement

  → If port move won't help, consider: TIP the FF (Step 0.7),
    spec rebalance (Step 0.8), or cell upsizing (Step 0.6)

  │ ⚠ RTL FIX RULES — WHAT WE DO AND DON'T DO                            │
  │                                                                        │
  │ When retiming/neighbor-stage analysis shows RTL4BE is needed:          │
  │                                                                        │
  │ ✅ ALLOWED:                                                            │
  │   • Retiming — move logic across existing FF boundary (no new FFs)    │
  │   • Reduce logic depth — simplify combinational cone (fewer gates)    │
  │   • Pre-compute — shift work to earlier cycle                         │
  │                                                                        │
  │ ❌ NEVER propose adding pipeline stages (new FFs to cut path):        │
  │   → Adds latency to design (e.g., +1 cycle to branch dispatch)       │
  │   → This is an architecture change, NOT a timing fix                  │
  │   → We fix timing by MOVING or REDUCING logic, never by ADDING FFs   │

================================================================================
  LIMITATIONS — WHAT pt_client CANNOT DO
================================================================================

  pt_client is READ-ONLY. For these tasks, you still need local PT or tools:

  × Cannot run ECO commands (size_cell, insert_buffer)
  × Cannot source custom TCL scripts
  × Cannot set attributes or create constraints
  × Cannot save sessions or export data
  × Cannot run carpet/SIO locally (but see carpet via pt_server — ask SIO)
  × Cannot check constraints/exceptions (report_exceptions)

  FOR THESE, use the traditional workflow:
    1. Navigate to WA sta_pt directory
    2. Open PT shell: pt_shell -f setup.tcl
    3. Load DBS: restore_session
    4. Run your commands
    5. Close: exit

  OR: Ask SIO to run carpet with pt_server (see ahaimovi's run_local_carpet)
